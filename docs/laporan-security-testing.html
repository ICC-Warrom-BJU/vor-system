04 Detail Temuan & Perbaikan

SEC-01Registrasi publik dengan role bebasCriticalA01

Deskripsi

Endpoint POST /api/auth/register terdaftar sebagai rute publik (tanpa middleware auth). Schema menerima field role (enum ADMIN|PLANNER|SUPERVISOR|MANAGEMENT) dan controller menyimpannya apa adanya (role: body.role).

Dampak

Siapa pun di internet dapat registrasi dengan "role":"ADMIN" dan langsung memperoleh akun administrator — eskalasi hak akses penuh, bypass seluruh RBAC. Aktif di produksi saat ditemukan.

Bukti (PoC)

POST /api/auth/register
{ "name":"Hacker", "email":"hack@evil.com", "password":"secret123", "role":"ADMIN" }
→ (sebelum fix) akun ADMIN terbentuk

Perbaikan

Endpoint publik /api/auth/register dihapus. Pembuatan user kini hanya via POST /api/users (dilindungi authMiddleware + roleGuard(['ADMIN'])). Frontend tidak memakai endpoint register (dikonfirmasi).

✅ Verifikasi produksi: POST /api/auth/register → HTTP 404 (endpoint tidak ada lagi)
SEC-02JWT_SECRET fallback hardcodedHighA02/A05

Deskripsi

Secret JWT memakai default hardcoded yang tercantum di kode sumber: process.env.JWT_SECRET || 'vor_super_secret_jwt_key_ganti_ini_nanti' — di auth.ts (sign) & middleware/auth.ts (verify).

Dampak

Bila JWT_SECRET lupa di-set, aplikasi memakai secret yang diketahui publik → penyerang bisa menempa (forge) token admin yang valid. "Landmine" konfigurasi.

Perbaikan

Modul config/env.ts memvalidasi env wajib saat startup & fail-fast bila kosong. Kedua fallback hardcoded dihapus.

✅ Verifikasi: start tanpa JWT_SECRET → FATAL: ... wajib di-set (server gagal start). Produksi sudah men-set secret → jalan normal; nilai tak diubah → sesi user existing tetap valid.
SEC-03Tidak ada rate-limit pada loginHighA07

Deskripsi

POST /api/auth/login tanpa pembatasan laju → brute-force password tak terbatas.

Perbaikan

Ditambahkan express-rate-limit: maks 10 percobaan / IP klien / 15 menit → HTTP 429. Karena Railway di belakang reverse proxy, req.ip tak stabil (menunjuk IP proxy internal yang berganti → penghitung reset); diperbaiki dengan keyGenerator yang mengambil IP klien dari entri paling kiri X-Forwarded-For + trust proxy.

✅ Verifikasi produksi: 12–14× login (IP sama) → req 1–10: 401, req ke-11+: 429. IP berbeda-beda tidak saling menghitung (kunci per-IP benar).
SEC-04Tidak ada HTTP security headersMediumA05

Deskripsi

Respons tanpa header keamanan standar (clickjacking, MIME-sniffing, HSTS, dll).

Perbaikan

Ditambahkan middleware helmet().

✅ Verifikasi produksi: Content-Security-Policy, Strict-Transport-Security, X-Content-Type-Options: nosniff, X-Frame-Options: SAMEORIGIN terpasang.
SEC-05CORS default ke *MediumA05

Deskripsi

origin: CORS_ORIGIN?.split(',') || '*' → bila env kosong, izinkan semua origin.

Perbaikan

Default * dihapus: ada CORS_ORIGIN → pakai daftarnya; kosong → produksi tolak cross-origin (dev diizinkan). Produksi sudah men-set CORS_ORIGIN.

SEC-06Kebocoran detail error internalMediumA05

Deskripsi

Middleware auth mengembalikan detail error internal: token invalid membalas error: <jwt message>; cabang 500 membalas error: String(error).

Perbaikan

Respons dibuat generik tanpa detail internal.

✅ Verifikasi produksi: token invalid → {"success":false,"message":"Token tidak valid"} (tanpa detail).
SEC-07Body request tanpa batas ukuranLowA05

Deskripsi

express.json()/urlencoded() tanpa limit → potensi large-payload DoS.

Perbaikan

Ditetapkan batas 1 MB untuk JSON & urlencoded.

05 Temuan Positif (kontrol yang sudah baik)

  • ✓
    RBAC ditegakkan di sisi server pada semua endpoint terlindungi (bukan hanya UI) — terverifikasi pada Functional Testing.
  • ✓
    Password di-hash dengan bcrypt (bukan plaintext).
  • ✓
    Perlindungan SQL Injection — seluruh akses DB via Prisma ORM (query terparameterisasi), tanpa raw SQL dari input.
  • ✓
    Audit log tidak membocorkan kredensial (field password disanitasi; login tidak dicatat).
  • ✓
    Validasi input via Zod (safeParse → 400) pada endpoint mutasi.
  • ✓
    Pesan login generik ("Email atau password salah") — mencegah user enumeration.
